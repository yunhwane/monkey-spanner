용어집 (Glossary)

이 코스의 약속된 어휘. 모든 레슨은 여기 정의를 따릅니다. 새 개념을 배울 때마다 갱신됩니다.

추상 abstraction
본질(무엇을 하는가, what)만 남기고 세부(어떻게 하는가, how)는 이름 뒤로 감추는 것. 클린코드·객체지향의 뿌리 개념. Lesson 01
추상화 수준 level of abstraction
코드가 이야기하는 "고도". "결제한다"는 높은 수준, "가격×수량을 더한다"는 낮은 수준. 한 함수 안의 문장들은 같은 수준에 있어야 읽힌다. Lesson 01
한 함수 한 수준 원칙 One Level of Abstraction per Function
함수 본문의 모든 문장이 함수 이름이 약속한 것과 같은 추상화 수준에 있어야 한다는 원칙. Clean Code Ch.3 · Lesson 01
메서드 추출 Extract Method
코드 덩어리를 의도를 드러내는 이름의 메서드로 뽑아내, 세부를 그 이름 뒤로 감추는 리팩토링 기법. Refactoring Guru · Lesson 01
코드 냄새 code smell
버그는 아니지만 "무언가 잘못됐다"는 신호를 주는 코드의 패턴. 이름을 붙일 수 있으면 고칠 수 있다. 예: Long Method(긴 메서드). Refactoring Guru · Lesson 01
리팩토링 refactoring
겉으로 드러나는 동작은 그대로 두고, 내부 구조를 더 읽기 좋게 바꾸는 것. Lesson 01
캡슐화 encapsulation
데이터와 그 데이터를 다루는 로직을 한 객체 안에 묶고, 바깥에는 행동만 노출하는 것. 추상을 객체 수준으로 확장한 것. Lesson 02
묻지 말고 시켜라 Tell, Don't Ask
객체의 속을 꺼내(ask) 바깥에서 판단하지 말고, 하고 싶은 일을 객체에게 시켜라(tell). 판단 로직을 데이터가 있는 곳에 둔다. Martin Fowler · Lesson 02
기능 편애 Feature Envy
어떤 코드가 자기 것보다 남(다른 객체)의 데이터를 더 많이 주무르는 냄새. 그 로직을 데이터가 있는 객체로 옮기라는 신호. Refactoring Guru · Lesson 02
불변식 invariant
객체가 항상 지켜야 하는 규칙(예: 잔액은 음수가 될 수 없다). 캡슐화로 바깥이 이를 깨지 못하게 지킨다. Lesson 02
빈약한 도메인 모델 anemic domain model
필드마다 getter/setter만 잔뜩 있고 행동은 없는 객체. private 이름표만 붙었을 뿐 캡슐화가 아니다. Lesson 02
다형성 polymorphism
같은 메서드 호출에, 객체의 실제 타입이 스스로 다른 행동을 고르게 하는 것. if/switch 분기를 타입에게 넘긴다. Lesson 03
동적 디스패치 dynamic dispatch
x.method() 호출 시, x의 런타임 실제 타입에 맞는 메서드가 선택되는 것. 다형성의 엔진. Lesson 03
조건문을 다형성으로 바꾸기 Replace Conditional with Polymorphism
타입 코드로 분기하는 switch/if 사슬을, 각 타입이 자기 행동을 갖는 클래스들로 바꾸는 리팩토링. Refactoring Guru · Lesson 03
개방-폐쇄 원칙 Open–Closed Principle (SOLID · O)
확장에는 열려, 수정에는 닫혀 있어야 한다. 새 기능은 새 코드 추가로 되어야지 기존 코드 수정이면 안 된다. 다형성이 이를 구현하는 대표 도구. Lesson 03
상속 inheritance
부모의 특성을 자식이 물려받는 것. "is-a(∼는 ∼의 일종)" 관계를 표현하는 도구이지, 코드 재사용 수단이 아니다. Lesson 04
조합 composition
다른 객체를 필드로 가지고(has-a) 필요한 것만 골라 쓰는 것. 재사용이 목적이면 상속보다 조합. Effective Java · Lesson 04
is-a / has-a 테스트 is-a / has-a
"A는 B의 일종인가(is-a → 상속 후보)" vs "A는 B를 가지는가/사용하는가(has-a → 조합)". 상속·조합 판별 기준. Lesson 04
리스코프 치환 원칙 Liskov Substitution Principle (SOLID · L)
부모 타입을 쓰는 코드에 자식을 넣어도 기대(약속)가 깨지지 않아야 한다. 깨지면 진짜 is-a가 아니다. Lesson 04
단일 책임 원칙 Single Responsibility Principle (SOLID · S)
한 클래스는 바뀌어야 하는 이유가 하나여야 한다. "책임"은 기능 개수가 아니라 변경의 축으로 센다. L01 "한 함수 한 수준"의 클래스 버전. Lesson 05
인터페이스 분리 원칙 Interface Segregation Principle (SOLID · I)
쓰지 않는 메서드에 의존하게 만들지 마라. 뚱뚱한 인터페이스 하나보다 작은 인터페이스 여럿. Lesson 05
의존성 역전 원칙 Dependency Inversion Principle (SOLID · D)
고수준(정책)이 저수준(세부)에 의존하지 말고, 둘 다 추상에 의존하라. 구체가 아니라 인터페이스에 의존 + 주입. Lesson 05
의존성 주입 Dependency Injection (DI)
객체가 필요한 협력자를 스스로 new 하지 않고 밖에서(생성자 등) 받는 기법. DIP를 실현하는 대표 수단. Lesson 05
함수형 인터페이스 functional interface
추상 메서드가 딱 하나인 인터페이스(@FunctionalInterface). 람다로 바로 구현 가능. 예: Comparator의 compare. Lesson 06
전략 패턴 Strategy Pattern
바뀌는 행동(알고리즘)을 인터페이스로 뽑아, 바깥에서 갈아끼우도록 주입하는 설계. 다형성+OCP+DIP의 결합. 예: List.sort(Comparator). Lesson 06
