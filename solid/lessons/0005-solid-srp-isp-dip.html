잘 읽히는 코드 · Lesson 05

SOLID 완성: S · I · D를 이미 배운 것으로 꿰기

새 법칙 3개가 아니라, 추상·캡슐화·조합의 다른 이름

SOLID의 O(L03)와 L(L04)은 이미 채웠습니다. 남은 S·I·D는 외울 새 규칙이 아니라, 지금까지 배운 걸 다른 각도에서 부르는 이름입니다. 공통 목표는 하나 — "변화가 한 곳에만 닿게, 나머지를 추상 뒤에 두라." 그게 SOLID 전체의 정신입니다.

한 줄 요약 S = 클래스는 바뀌는 이유가 하나  ·  I = 안 쓰는 메서드에 의존하지 마라  ·  D = 구체가 아니라 추상에 의존하라.

S — 단일 책임 원칙 (Single Responsibility)

Lesson 01의 "한 함수는 한 수준"을 클래스 레벨로 올린 것입니다. "한 클래스는 바뀌는 이유가 하나여야 한다." 아래 클래스가 바뀔 이유를 세어봅시다.

Before — 바뀔 이유가 셋
class UserService {
  void register(Form f) {
    validate(f);          // ① 검증 규칙
    db.save(toUser(f));   // ② DB 스키마
    mail.send(...);       // ③ 메일 형식
  }
}
After — 각자 이유 하나씩
class SignupValidator { ... }  // ①만
class UserRepository { ... }    // ②만
class WelcomeMailer { ... }     // ③만

class UserService {           // 조립만
  ... register() { v...; r...; m...; }
}

검증 규칙이 바뀌어도 DB 코드는 안 건드립니다. 각 클래스가 한 가지 이유로만 바뀌니까요. "책임"은 기능의 개수가 아니라 "바뀌는 이유(변경의 축)"로 세는 게 핵심입니다.

I — 인터페이스 분리 원칙 (Interface Segregation)

Lesson 04에서 Stack이 리스트의 모든 메서드에 노출돼 곤란했죠. 그 문제의 인터페이스 버전입니다. "쓰지도 않는 메서드에 의존하지 마라. 큰 인터페이스 하나보다 작은 것 여럿."

Before — 뚱뚱한 인터페이스
interface Machine {
  void print();
  void scan();
  void fax();
}
// 단순 프린터도 scan/fax를
// 억지로 구현해야 함(빈 메서드…)
class SimplePrinter
      implements Machine { ... }
After — 잘게 나눈 인터페이스
interface Printer { void print(); }
interface Scanner { void scan(); }
interface Fax     { void fax(); }

class SimplePrinter
      implements Printer { ... }
class AllInOne
  implements Printer, Scanner, Fax { ... }

필요한 능력만 골라 구현합니다. 빈 fax(){} 같은 거짓 구현이 사라지고, 각 클라이언트는 자기가 쓰는 계약에만 묶입니다. Lesson 04의 "노출을 최소화하라"와 같은 정신 — 이번엔 계약(인터페이스)에 적용한 거죠.

D — 의존성 역전 원칙 (Dependency Inversion)

Lesson 04에서 EmailClient를 주입받았던 그 조합 — 사실 그게 D의 씨앗이었습니다. "고수준(정책)이 저수준(세부)에 의존하지 말고, 둘 다 추상에 의존하라."

Before — 구체에 직접 의존
class NotificationService {
  // EmailClient에 못박힘
  private final EmailClient email
      = new EmailClient();
  void notify(User u, String m) {
    email.send(u.getEmail(), m);
  }
}
// SMS로 바꾸려면 이 클래스를 수정
After — 추상에 의존 + 주입
interface Notifier {         // 추상
  void send(User u, String m);
}
class NotificationService {
  private final Notifier notifier;
  NotificationService(Notifier n) {  // 주입
    this.notifier = n;
  }
  void notify(User u, String m) {
    notifier.send(u, m);
  }
}

이제 NotificationService(고수준 정책)는 EmailNotifier인지 SmsNotifier인지 모릅니다. 둘 다 Notifier라는 추상만 바라보죠. 이게 L03의 OCP와 손잡습니다 — 새 알림 수단을 추가해도 서비스는 안 바뀝니다. "역전"이란, 원래 서비스→이메일로 향하던 의존의 화살표를, 이메일→인터페이스←서비스 로 뒤집어 세부가 추상을 향하게 만든 것.

전체를 하나로 SOLID 다섯 글자는 결국 "변경이 번지지 않도록, 추상 뒤에서 의존을 관리하라"는 한 생각의 다섯 얼굴입니다. S로 이유를 나누고, O·L·I·D로 그 경계를 추상으로 안전하게 잇는 거죠.

확인해볼까요

단일 책임 원칙에서 "책임이 하나"라는 말을 가장 정확히 옮긴 것은?

핵심: SRP의 "책임"은 메서드/필드 개수가 아니라 바뀌는 이유입니다. 검증·DB·메일은 각기 다른 이유로 바뀌므로 서로 다른 책임 → 다른 클래스로.

NotificationService가 Notifier 인터페이스에 의존하게 만든 이득은? (DIP)

핵심: 고수준 정책이 추상에만 의존하니, 저수준 세부(Email/SMS)를 갈아끼워도 정책은 그대로입니다. DIP는 OCP와 짝을 이뤄 "확장에 열리고 수정에 닫힌" 구조를 만듭니다.

4분 실습 — 당신 차례

아래 OrderService는 SOLID를 여러 개 어깁니다. 어떤 글자를 어기고 있는지 짚고, 어떻게 나눌지(어떤 클래스/인터페이스로) 스케치해 저에게 보여주세요.

class OrderService {
  private MySqlDatabase db = new MySqlDatabase();  // 구체 클래스에 직접 의존
  void placeOrder(Order o) {
    if (o.getItems().isEmpty()) throw new InvalidOrder();  // 검증
    db.save(o);                                          // 저장
    String csv = o.getId() + "," + o.getTotal();          // 리포트 포맷
    reportFile.append(csv);                              // 파일 쓰기
    mail.send(o.getBuyerEmail(), "주문 완료");             // 메일
  }
}

힌트: 바뀌는 이유가 몇 개인가요?(검증/저장/리포트/메일 → S) 그리고 new MySqlDatabase()처럼 구체에 못박힌 곳은?(→ D: OrderRepository 인터페이스로 추상화 후 주입)
