잘 읽히는 코드 · Lesson 01

추상(抽象): 코드가 "읽히게" 만드는 첫 번째 힘

"와, 이 사람 코드 잘 짠다"의 정체를, 함수 하나로 체감하기

우리가 "코드를 잘 짠다"고 느낄 때, 사실은 "잘 읽힌다"고 느끼는 겁니다. 그리고 코드가 잘 읽히는지 아닌지를 결정하는 가장 밑바닥 개념이 바로 추상(abstraction)입니다. 클린코드도, 객체지향도, SOLID도 전부 이 한 뿌리에서 자랍니다. 이번 레슨에서 그 뿌리를 손으로 만져봅니다.

Key idea 추상이란 "무엇을 하는가(what)"만 남기고, "어떻게 하는가(how)"는 이름 뒤로 감추는 것. 좋은 함수 이름은 본문을 안 봐도 이해되고, 궁금할 때만 안으로 내려가 보게 만듭니다.

당신은 이미 매일 추상을 쓴다

"커피 한 잔 타 줘"라고 말할 때, 우리는 물을 끓이고, 원두를 갈고, 드리퍼에 붓고…를 일일이 말하지 않습니다. "커피를 탄다"는 이름 하나가 그 모든 세부를 감춰줍니다. 듣는 사람은 세부를 몰라도 뭘 하려는지 압니다. 이게 추상입니다. 코드도 똑같아야 합니다 — 읽는 사람이 한 번에 한 수준만 생각하게.

냄새나는 코드: 모든 수준이 뒤섞여 있다

주문을 결제하는 Java 메서드입니다. 한 번 읽어 내려가 보세요. 몇 초 만에 "무슨 일을 하는지" 파악되나요?

void checkout(Cart cart, User user) {
  // 합계 계산
  int total = 0;
  for (Item i : cart.getItems()) {
    total += i.getPrice() * i.getQty();
  }
  // 회원 등급 할인
  if (user.getGrade().equals("VIP")) total = total * 90 / 100;
  // 카드 결제 호출
  PgResponse res = pg.request(user.getCardNo(), total, "KRW");
  if (!res.getCode().equals("0000")) throw new PayException();
  // 영수증 문자열 조립 후 발송
  String body = "결제금액: " + total + "원";
  mail.send(user.getEmail(), "영수증", body);
}

읽기 힘든 이유는 어렵기 때문이 아닙니다. 수준이 뒤섞였기 때문입니다. "결제한다"라는 높은 수준의 이야기 사이에, total += i.getPrice() * i.getQty() 같은 낮은 수준의 계산 디테일이 끼어 있습니다. 읽는 사람의 머리는 매 줄마다 고도를 오르내려야 합니다. 그게 피로의 정체입니다.

코드 냄새 이름: 이건 Long Method(긴 메서드)이자, Uncle Bob이 말한 "한 함수 안에 여러 추상화 수준이 섞임"의 전형입니다. 이름을 붙일 수 있다는 건, 고칠 수 있다는 뜻입니다.

리팩토링: 세부를 이름 뒤로 감추기 (Extract Method)

각 "낮은 수준" 덩어리를 의도를 드러내는 이름의 메서드로 뽑아냅니다. 그러면 checkout은 세부가 사라지고 이야기의 줄거리만 남습니다.

After — 줄거리만 남은 checkout
void checkout(Cart cart, User user) {
  int total = totalPrice(cart);
  total = applyGradeDiscount(total, user);
  pay(user, total);
  sendReceipt(user, total);
}
감춰진 세부 (궁금할 때만 열어본다)
private int totalPrice(Cart cart) {
  int total = 0;
  for (Item i : cart.getItems())
    total += i.getPrice() * i.getQty();
  return total;
}

private boolean isVip(User u) {
  return u.getGrade().equals("VIP");
}

이제 checkout은 네 줄의 한글 문장처럼 읽힙니다: 합계를 구하고 → 등급 할인을 적용하고 → 결제하고 → 영수증을 보낸다. 모든 줄이 같은 높이에 있습니다. 세부가 궁금하면 그때 totalPrice 안으로 한 층 내려가면 됩니다. 이것이 바로—

원칙 한 함수는 한 수준의 추상화만 담는다 (One Level of Abstraction per Function). 함수 본문의 모든 문장이 이름이 약속한 것과 같은 고도에 있어야 합니다.

확인해볼까요

리팩토링한 checkout에 아래 한 줄을 그대로 넣으면 왜 어색할까요?
total += item.getPrice() * item.getQty();

핵심: checkout의 다른 줄은 "합계·할인·결제·영수증" 같은 줄거리 수준인데, 이 줄만 곱셈 루프 수준의 세부입니다. 고도가 어긋나면 읽는 사람이 다시 오르내려야 하죠. 이 세부는 totalPrice() 이름 뒤에 감춰져 있어야 합니다.

추출한 메서드 이름으로 가장 좋은 것은? (모두 같은 동작을 한다고 가정)

핵심: 좋은 이름은 무엇을 하는지를 딱 필요한 만큼 드러냅니다. applyGradeDiscount는 의도가 분명하고 checkout의 다른 줄과 같은 높이입니다. 너무 장황한 이름(0번)이나 의미 없는 이름(1번)은 오히려 추상을 망칩니다.

1분 실습 — 당신 차례

아래 메서드를 눈으로 리팩토링해 보세요. 어느 줄이 "너무 낮은 수준"이라 이름 뒤로 감춰야 할까요? 어떤 이름의 메서드로 뽑을지 종이나 주석으로 적어보고, 저(선생님)에게 결과를 보여주세요. 

void register(SignupForm form) {
  if (!form.getEmail().contains("@")) throw new InvalidEmail();
  if (form.getPassword().length() < 8) throw new WeakPassword();
  String hash = sha256(form.getPassword() + SALT);
  userRepo.save(new User(form.getEmail(), hash));
  mail.send(form.getEmail(), "환영합니다", welcomeBody());
}

힌트: "검증"과 "비밀번호 해싱"은 각각 이름 하나로 감출 수 있는 세부입니다. 목표 모습 → validate(form); User user = createUser(form); userRepo.save(user); sendWelcome(user);
