잘 읽히는 코드 · Lesson 03

다형성: 분기를 없애고 타입이 스스로 고르게

if/else if 사슬을 지우면, SOLID의 'O'가 보인다

Lesson 02에서 판단을 객체 안으로 옮겼습니다. 그런데 아직 이런 코드가 남아 있죠 — if (grade == VIP) … else if (grade == GOLD) …. 다형성은 이 분기 자체를 지웁니다. "무엇을 할지 고르는 일"을 if가 아니라 타입(객체)에게 맡기는 거예요.

Key idea 다형성 = 같은 메서드 호출에, 객체의 실제 타입이 스스로 다른 행동을 고르게 하는 것. 분기(if/switch)로 "어떤 종류냐"를 묻던 것을, 타입이 대신 결정합니다.

냄새: 똑같은 분기가 여기저기 복제된다

결제 수단마다 다르게 동작하는 코드입니다. switch(또는 if 사슬)가 메서드마다 반복됩니다.

int fee(String type, int amount) {
  if (type.equals("CARD"))  return amount * 2 / 100;
  if (type.equals("KAKAO")) return amount * 1 / 100;
  if (type.equals("BANK"))  return 0;
  throw new IllegalArgumentException();
}

String label(String type) {
  if (type.equals("CARD"))  return "신용카드";
  if (type.equals("KAKAO")) return "카카오페이";
  if (type.equals("BANK"))  return "계좌이체";
  throw new IllegalArgumentException();
}
코드 냄새 이름: Switch Statements(같은 타입 코드로 분기하는 switch/if가 반복됨). 결제 수단 하나(예: "PAYPAL")를 추가하면 이 모든 분기를 찾아 똑같이 고쳐야 합니다. 하나라도 빠뜨리면 버그죠. Lesson 02의 Feature Envy와 뿌리가 같습니다 — 지식이 흩어졌다.

리팩토링: 타입마다 자기 행동을 갖게 하라

"CARD가 무엇을 하는가"를 CardPayment라는 타입 안에 담습니다. 공통 약속은 interface로 정합니다. 

interface PaymentMethod {
  int fee(int amount);
  String label();
}

class CardPayment implements PaymentMethod {
  public int fee(int amount) { return amount * 2 / 100; }
  public String label() { return "신용카드"; }
}
class KakaoPayment implements PaymentMethod {
  public int fee(int amount) { return amount * 1 / 100; }
  public String label() { return "카카오페이"; }
}

이제 분기가 통째로 사라집니다. 부르는 쪽은 어떤 타입인지 묻지 않고 그냥 시킵니다:

Before — 부르는 쪽이 분기한다
int f = fee(type, amount);
String name = label(type);
After — 타입이 스스로 고른다
int f = method.fee(amount);
String name = method.label();

method.fee(amount) 한 줄이 실행될 때, method의 실제 타입(CardPayment인지 KakaoPayment인지)이 어떤 fee를 부를지 런타임에 스스로 결정합니다. 이걸 동적 디스패치(dynamic dispatch)라 하고, 이게 다형성의 엔진입니다. "어떤 종류냐"를 우리가 if로 묻던 일을, 언어가 대신 해주는 거죠.

여기서 SOLID의 'O'가 태어난다 — 개방-폐쇄 원칙

이제 "PAYPAL"을 추가해 봅시다. 무엇을 고쳐야 할까요?

// 새 파일 하나를 "추가"할 뿐, 기존 코드는 한 줄도 "수정"하지 않는다
class PaypalPayment implements PaymentMethod {
  public int fee(int amount) { return amount * 3 / 100; }
  public String label() { return "페이팔"; }
}
SOLID · O 개방-폐쇄 원칙(Open–Closed Principle): 확장에는 열려 있고, 수정에는 닫혀 있어야 한다. 새 기능은 새 코드를 더하는 것으로 되어야지, 잘 돌던 기존 코드를 뜯어고치는 것이면 안 된다.

switch 버전에선 결제 수단 하나 추가에 fee, label, process… 모든 분기를 수정해야 했습니다(닫혀 있지 않음). 다형성 버전에선 클래스 하나를 추가하고 끝입니다. 기존 코드는 건드리지 않으니 기존 기능이 깨질 위험이 0이죠. 다형성은 OCP를 구현하는 가장 흔한 도구입니다.

과용 주의: 분기가 딱 한 곳에만 있고 늘어날 일이 없다면, 굳이 클래스로 쪼개지 마세요. 다형성은 "같은 종류의 분기가 여러 곳에 반복되거나, 종류가 계속 늘어날 때" 값을 합니다. 반대로, 밖에서 if (x instanceof CardPayment)로 타입을 되묻고 있다면 — 다형성을 놓친 신호입니다.

확인해볼까요

결제 수단 PAYPAL 추가 시, 다형성 버전이 switch 버전보다 나은 핵심 이유는?

핵심: 이게 개방-폐쇄 원칙입니다. 잘 돌던 fee/label을 안 건드리므로, 기존 결제 수단이 깨질 위험 없이 확장됩니다. 흩어진 switch를 다 고칠 때 생기는 "빠뜨림 버그"가 사라지죠.

다음 중 "다형성을 놓친" 냄새는?

핵심: 2번은 타입을 다시 instanceof로 묻고 분기합니다 — switch를 이름만 바꿔 되살린 셈이죠. 그 분기 안의 행동을 각 클래스의 메서드로 옮기면 m.something() 한 줄로 사라집니다.

3분 실습 — 당신 차례

아래 알림(Notification) 코드를 다형성으로 바꿔보세요. 어떤 인터페이스와 클래스들로 나눌지, 그리고 부르는 쪽 코드가 어떻게 짧아지는지 스케치해서 저에게 보여주세요.

void notify(String channel, User user, String msg) {
  if (channel.equals("EMAIL")) mail.send(user.getEmail(), msg);
  else if (channel.equals("SMS"))  sms.send(user.getPhone(), msg);
  else if (channel.equals("PUSH")) push.send(user.getDeviceId(), msg);
}

힌트: 공통 약속은 interface Notifier { void send(User user, String msg); }. 목표 느낌 → 부르는 쪽은 notifier.send(user, msg); 한 줄. 새 채널(예: 카카오톡)은 클래스 하나 추가로 끝나야 합니다(OCP).
