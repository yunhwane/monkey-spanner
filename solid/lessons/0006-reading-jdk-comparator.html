잘 읽히는 코드 · Lesson 06

실전 오픈소스 분석: JDK의 Comparator를 읽다

지어낸 예제는 끝. 수억 명이 쓰는 코드를 당신의 렌즈로

지금까지의 예제는 제가 만든 거였습니다. 이번엔 실제 자바 표준 라이브러리를 읽습니다. 놀라운 건 — java.util.Comparator 하나에 우리가 배운 렌즈가 거의 다 들어 있다는 점이에요. 좋은 코드는 이렇게 원칙들이 자연스레 겹쳐 나타납니다. 하나씩 짚어봅시다.

오늘의 목표 실제 설계를 보고 "이 사람들은 왜 이렇게 짰을까"를 배운 용어로 스스로 해설하기. 이게 면접에서 "코드 볼 줄 안다"를 증명하는 능력입니다.

① 핵심은 딱 한 줄이다 — 작은 인터페이스 (ISP)

Comparator의 본질은 이 한 줄입니다 (출처: Oracle 공식 API 문서): 

@FunctionalInterface
public interface Comparator<T> {
  int compare(T o1, T o2);   // 유일한 추상 메서드
}

"두 값을 비교한다"라는 단 하나의 책임. 이게 Lesson 05의 ISP·SRP입니다 — 인터페이스가 딱 필요한 만큼만 작죠. 그래서 @FunctionalInterface (추상 메서드 하나짜리)로 성립하고, 람다로 바로 쓸 수 있습니다: 

Comparator<User> byAge = (a, b) -> a.getAge() - b.getAge();   // 인터페이스 = 람다 한 줄

② sort는 "비교 규칙"을 주입받는다 — DIP + OCP + 전략 패턴

이제 진짜 배울 점입니다. List.sort를 봅시다:

List<User> users = ...;
users.sort(byAge);                          // 나이순
users.sort((a, b) -> a.getName().compareTo(b.getName()));  // 이름순

sort의 정렬 알고리즘(TimSort)은 단 한 줄도 바뀌지 않습니다. 그런데 정렬 기준은 무한히 바뀔 수 있죠. 어떻게? 비교 규칙을 바깥에서 주입받기 때문입니다.

  • DIP(L05): 고수준 정렬 알고리즘이 구체적 비교 로직이 아니라 Comparator라는 추상에만 의존합니다.
  • OCP(L03): 새 정렬 기준이 필요하면 Comparator를 추가할 뿐, sort를 수정하지 않습니다.
  • 다형성: sort 안의 compare(a, b) 호출이, 주입된 실제 Comparator의 행동을 런타임에 고릅니다.
패턴 이름: "행동(알고리즘)을 인터페이스로 뽑아 바깥에서 갈아끼우는" 이 설계를 전략 패턴(Strategy Pattern)이라 합니다. Lesson 03에서 PaymentMethod를 갈아끼운 것과 똑같은 구조예요 — 당신은 이미 GoF 디자인 패턴 하나를 알고 있었던 겁니다.

③ 작은 비교자를 조립해 큰 비교자를 만든다 — 조합 (Composition)

"이름순, 같으면 나이순, 전체를 역순으로." 이걸 자바는 거대한 if가 아니라 조합으로 표현합니다: 

Comparator<User> order =
    Comparator.comparing(User::getName)   // 1차 기준
              .thenComparing(User::getAge) // 동점이면 2차 기준
              .reversed();                 // 전체 역순
users.sort(order);

thenComparing·reversed는 기존 Comparator를 받아 새 Comparator를 돌려줍니다. 작은 부품을 가져다(has-a) 더 큰 걸 만드는 Lesson 04의 조합이죠. 공식 소스에서 thenComparing의 본질은 이렇습니다 (Oracle 문서 기준, 요지): 

default Comparator<T> thenComparing(Comparator<? super T> other) {
  return (a, b) -> {
    int res = this.compare(a, b);        // 나를 먼저 써 보고
    return (res != 0) ? res : other.compare(a, b);  // 동점이면 other에게 위임
  };
}

문서의 표현 그대로: "이 Comparator가 두 원소를 같다고 보면(compare(a,b)==0), other가 순서를 정한다." 상속(extends)은 한 군데도 없이, 조합만으로 무한한 정렬 규칙을 만들어냅니다. 이게 Effective Java가 "상속보다 조합"이라 한 이유의 살아있는 증거예요.

④ 반례로 배우기: Stack은 왜 실패작인가 (상속·LSP)

같은 java.util인데 정반대인 코드가 있습니다. Lesson 04에서 예고한 그것 — 공식 문서가 스스로 인정한 설계 실수입니다. Oracle 문서 원문: 

public class Stack<E> extends Vector<E>   // 재사용하려고 Vector를 상속
공식 문서의 권고 (원문 인용): "A more complete and consistent set of LIFO stack operations is provided by the Deque interface and its implementations, which should be used in preference to this class. For example: Deque<Integer> stack = new ArrayDeque<Integer>();"
→ "새 코드는 Stack 대신 Deque/ArrayDeque를 써라." 표준 라이브러리가 자기 클래스를 말리는 겁니다.

이유는 우리가 이미 압니다: Stack extends Vector라서 stack.add(0, x)(맨 아래 삽입), stack.remove(1)(중간 제거) 같은 스택이 아닌 연산이 다 노출됩니다. "스택은 맨 위로만"이라는 약속이 깨지죠 — 잘못된 상속 → 캡슐화·LSP 붕괴. Comparator(조합)와 Stack(상속)을 나란히 놓으면, 우리가 배운 원칙이 왜 옳은지가 실물로 증명됩니다.

확인해볼까요

users.sort(comparator)에서, 새 정렬 기준을 추가해도 sort 알고리즘 코드가 안 바뀌는 것은 어떤 원칙인가?

핵심: 정렬 기준(Comparator)을 추가할 뿐 sort를 수정하지 않으니 OCP. 그리고 규칙을 추상으로 주입받으니 DIP가 짝을 이루죠. 이 조합이 "전략 패턴"입니다.

comparing(...).thenComparing(...).reversed()가 보여주는 설계 원리는?

핵심: 각 메서드는 기존 Comparator를 받아 새 것을 반환합니다(has-a 조합). 상속도, 분기 나열도 아니죠. "상속보다 조합"이 실제 API에서 어떻게 빛나는지의 표본입니다.

5분 실습 — 당신 차례 (실전 리딩)

아래는 실제로 자주 보는 코드입니다. 이 한 줄에 우리가 배운 원칙/개념이 몇 개나 겹쳐 있는지 찾아서, 각각을 이름 붙여 저에게 설명해 주세요. (힌트: 최소 3개는 보입니다.) 

employees.sort(
    Comparator.comparing(Employee::getDept)
              .thenComparing(Employee::getSalary, Comparator.reverseOrder())
);

생각해볼 질문: (1) sort는 정렬 규칙을 어떻게 아나요? (2) getSalary 규칙과 getDept 규칙이 어떻게 하나로 합쳐지나요? (3) 부서별 정렬을 "이름별"로 바꾸려면 sort 내부를 고쳐야 하나요? — 각 답에 원칙 이름을 붙여보세요.
