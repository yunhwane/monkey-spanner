상속 vs 조합: 언제 extends가 독이 되는가

OOP 4번째 축, 그리고 SOLID의 'L'

Lesson 03에서 다형성으로 분기를 지웠습니다. 그런데 다형성을 쓰려면 보통 상속(또는 인터페이스 구현)이 필요하죠. 상속은 OOP의 4번째 축이자, 가장 오해와 남용이 많은 도구입니다. 이번엔 "상속을 언제 써야 하고, 언제 독이 되는가"를 가릅니다.

Key idea 상속은 "코드 재사용" 도구가 아니라 "is-a(∼는 ∼의 일종이다) 관계"를 표현하는 도구다. 재사용이 목적이라면 대개 조합(composition, has-a)이 정답이다.

달콤한 함정: 재사용하려고 상속한다

Lesson 03의 알림을 보내는 서비스를 만든다고 합시다. 이메일 보내는 EmailClient가 이미 있으니, "그 send()를 재사용하자"며 이렇게 씁니다: 

// 재사용이 탐나서 extends 해버림
class NotificationService extends EmailClient {
  void notifyUser(User u, String msg) {
    send(u.getEmail(), msg);   // 부모의 send() 재사용!
  }
}

돌아는 갑니다. 하지만 NotificationService는 이제 EmailClient의 모든 것을 물려받습니다:

  • service.connectSmtp(), service.setPort(587) 같은 내부용 메서드가 바깥에 노출됩니다.
  • "알림 서비스가 SMTP 포트를 갖는다"는 건 말이 안 됩니다 — 개념이 새어 나왔죠.
  • 부모 EmailClient의 구현이 바뀌면 자식이 조용히 깨질 수 있습니다(강한 결합).
판별 테스트 — is-a인가, has-a인가? "NotificationService 는 EmailClient 의 일종이다"? 어색하죠. 옳은 관계는 "NotificationService 는 EmailClient 를 가진다(사용한다)" — has-a입니다. has-a면 상속이 아니라 조합입니다.

리팩토링: 물려받지 말고, 가져서 쓴다 (Composition)

Before — 상속 (is-a 거짓)
class NotificationService
      extends EmailClient {
  void notifyUser(User u, String m) {
    send(u.getEmail(), m);
  }
}
// 바깥에서 이런 게 다 보임
service.connectSmtp();  // ?!
After — 조합 (has-a)
class NotificationService {
  private final EmailClient email;
  NotificationService(EmailClient e) {
    this.email = e;
  }
  void notifyUser(User u, String m) {
    email.send(u.getEmail(), m);
  }
}

조합 버전은 EmailClient를 필드로 가지고, 필요한 것(send)만 골라 씁니다. SMTP 세부는 새어 나오지 않고, 바깥에 노출되는 건 notifyUser 하나뿐입니다(캡슐화!). 게다가 생성자로 EmailClient를 주입받으니, 나중에 가짜(Mock)로 바꿔 테스트하기도 쉽죠.

상속이 정당할 때 지켜야 할 것: 리스코프 치환 원칙 (SOLID · L)

그럼 진짜 is-a일 땐? 그때도 규칙이 있습니다. 자식은 부모 자리에 넣어도 아무 문제가 없어야 합니다. 그 유명한 반례 — 정사각형은 직사각형의 일종(is-a)처럼 보이지만… 

class Rectangle {
  protected int w, h;
  void setWidth(int w)  { this.w = w; }
  void setHeight(int h) { this.h = h; }
  int area() { return w * h; }
}

class Square extends Rectangle {
  void setWidth(int s)  { this.w = this.h = s; }  // 정사각형이니 같이 바꿈
  void setHeight(int s) { this.w = this.h = s; }
}

// 부모를 기대하고 짠 코드
void resizeAndCheck(Rectangle r) {
  r.setWidth(5);
  r.setHeight(4);
  assert r.area() == 20;   // Rectangle이면 참. Square를 넣으면? area()==16 → 깨진다!
}
SOLID · L 리스코프 치환 원칙(Liskov Substitution Principle): 부모 타입을 쓰는 코드에 자식을 넣어도 기대가 깨지지 않아야 한다. 깨진다면 그건 진짜 is-a 관계가 아니다 — 상속이 잘못된 것.

Square는 "가로·세로를 독립적으로 바꿀 수 있다"는 Rectangle의 약속을 어깁니다. 그래서 Rectangle을 기대한 resizeAndCheck가 Square 앞에서 깨지죠. 교훈: "현실에서 A는 B의 일종"이라고 코드에서도 상속이 옳은 건 아니다. 행동(약속)까지 대체 가능해야 합니다.

실무 원칙 (Effective Java): "상속보다 조합을 favor하라." 상속은 (1) 진짜 is-a이고 (2) 자식이 부모를 완전히 대체할 수 있을 때만. 애매하면 조합입니다. 다형성이 필요할 뿐이라면 구현 상속(extends 클래스)보다 인터페이스가 대개 더 안전합니다(Lesson 03의 방식).

확인해볼까요

class NotificationService extends EmailClient의 가장 큰 문제는?

핵심: "알림서비스는 이메일클라이언트의 일종"이 아니라 "이메일클라이언트를 사용한다(has-a)"죠. 잘못된 상속은 SMTP 같은 내부를 바깥에 노출하고 부모 변경에 자식을 취약하게 만듭니다. 조합이 정답.

Square extends Rectangle가 리스코프 치환 원칙을 어긴 이유는?

핵심: LSP는 "자식을 부모 자리에 넣어도 기대가 유지되는가"입니다. Square는 setWidth가 height까지 바꿔버려 Rectangle의 약속을 깹니다. 넓이 공식이 아니라 대체 가능성이 핵심.

3분 실습 — 당신 차례

아래 설계가 상속으로 되어 있습니다. 상속이 옳은가? 아니라면 조합으로 어떻게 바꿀지, 그리고 그 근거(is-a / has-a 테스트)를 한 줄로 저에게 설명해 주세요.

// 스택을 만들려고 ArrayList를 상속함
class Stack<T> extends ArrayList<T> {
  void push(T item) { add(item); }
  T pop() { return remove(size() - 1); }
}
// 문제: 이렇게 하면 add(0, x), get(3), remove(1) ... 리스트의 모든 메서드가 스택에 노출됨

힌트: "Stack 은 ArrayList 의 일종인가?" 스택은 맨 위로만 넣고 뺀다는 약속이 있는데, ArrayList를 상속하면 아무 위치나 접근하는 메서드가 다 열려 그 약속이 깨집니다. 목표 느낌 → class Stack<T> { private final List<T> items = new ArrayList<>(); ... }
