잘 읽히는 코드 · Lesson 02

캡슐화: 데이터와 로직을 객체 뒤로 감추기

추상을 함수에서 객체로 확장하기 — "묻지 말고 시켜라"

Lesson 01에서 우리는 세부(how)를 함수 이름 뒤로 감췄습니다. 캡슐화는 같은 아이디어를 한 단계 키운 것입니다 — 이번엔 데이터와, 그 데이터를 다루는 로직을 객체 이름 뒤로 함께 감춥니다.

Key idea 캡슐화 = "데이터 + 그 데이터를 다루는 로직"을 한 객체 안에 묶고, 바깥에는 행동만 노출하는 것. 객체에게 속을 묻지(ask) 말고, 하고 싶은 일을 시켜라(tell).

다시 등장한 그 코드 — 무엇이 냄새일까?

Lesson 01의 checkout에 있던 이 줄들을 봅시다. 문법은 멀쩡한데, 뭔가 불편합니다.

// 여기저기 흩어져 나타나는 코드
if (user.getGrade().equals("VIP")) total = total * 90 / 100;
...
if (user.getGrade().equals("VIP")) showFreeShippingBadge();
...
if (user.getGrade().equals("VIP") || user.getGrade().equals("GOLD")) ...

이 코드가 바깥에서 저지르는 일을 세어봅시다:

  • getGrade()로 User의 속을 꺼내 옵니다. (묻는다 / ask)
  • "VIP"라는 문자열 규칙을 바깥이 알고 있습니다. (grade가 String이라는 것도)
  • "VIP란 무엇인가"라는 판단 로직이 User 밖에 흩어져 있습니다.
코드 냄새 이름: Feature Envy(기능 편애) — 바깥 코드가 남(User)의 데이터를 부러워하며 자기 일처럼 주무릅니다. 등급 체계가 바뀌면(예: VIP→PLATINUM) 이 getGrade().equals("VIP")를 전부 찾아 고쳐야 합니다. 이게 캡슐화가 깨졌을 때 치르는 비용입니다.

원칙: 묻지 말고 시켜라 (Tell, Don't Ask)

데이터를 꺼내 바깥에서 판단하지 말고, 판단을 데이터가 있는 곳(User)으로 가져가 이름을 붙입니다. "VIP란 무엇인가"는 User가 제일 잘 아는 사실이니까요.

Before — 묻는다 (ask)
// 바깥이 grade의 정체를 안다
if (user.getGrade()
        .equals("VIP")) {
  total = total * 90 / 100;
}
After — 시킨다 (tell)
// User 안으로 판단이 들어감
class User {
  private Grade grade;
  public boolean isVip() {
    return grade == Grade.VIP;
  }
}

// 바깥은 이제 이렇게만 읽힌다
if (user.isVip()) total = total * 90 / 100;

바뀐 것을 보세요. 이제 바깥은 "VIP"가 문자열인지 enum인지 모릅니다. 알 필요도 없죠. 등급 규칙이 바뀌면 User 한 곳만 고치면 됩니다. 세부(grade의 저장 방식, 비교 방법)가 isVip()라는 이름 뒤로 감춰진 겁니다 — Lesson 01과 정확히 같은 원리, 대상만 객체로 커졌습니다.

한 걸음 더: 할인 규칙까지 데이터 곁으로

"등급별 할인율"이라는 지식도 사실 User(혹은 Grade)의 것입니다. 여기까지 옮기면:

enum Grade {
  VIP(10), GOLD(5), NORMAL(0);
  private final int discountPercent;
  Grade(int p) { this.discountPercent = p; }
  public int discountPercent() { return discountPercent; }
}

// checkout은 이제 규칙을 모른 채, 시키기만 한다
total = total * (100 - user.getGrade().discountPercent()) / 100;

캡슐화의 다른 얼굴: 불변식 지키기

캡슐화는 "감추기"인 동시에 "망가뜨릴 수 없게 지키기"입니다. 필드를 public으로 열면, 누구든 객체를 말이 안 되는 상태로 만들 수 있습니다.

Before — 속이 다 열려 있다
class Account {
  public int balance;
}
// 바깥에서 이런 짓이 가능
account.balance = -99999;  // ?!
After — 규칙을 객체가 지킨다
class Account {
  private int balance;
  public void withdraw(int amount) {
    if (amount > balance)
      throw new IllegalStateException();
    balance -= amount;
  }
}
주의 — getter/setter를 다 열면 캡슐화가 아니다. 모든 필드에 getX()/setX()를 기계적으로 다는 건, private이라는 이름표만 붙였을 뿐 속을 다 여는 것과 같습니다(이를 빈약한 도메인 모델, anemic model이라 부릅니다). 진짜 캡슐화는 행동(behavior)을 노출하는 것 — getBalance()보다 withdraw()가 캡슐화입니다.

확인해볼까요

user.getGrade().equals("VIP")가 코드 곳곳에 흩어진 것이 왜 문제일까요?

핵심: "VIP란 무엇인가"라는 판단이 User 밖에 복제돼 있으면, 규칙 한 번 바뀔 때 모든 사본을 수정해야 합니다. isVip()로 모으면 바뀌는 이유가 한 곳에 있게 되죠 — 이게 캡슐화가 주는 유지보수성입니다.

"묻지 말고 시켜라(Tell, Don't Ask)"를 가장 잘 지킨 코드는?

핵심: 1·2번은 balance를 꺼내 바깥에서 계산합니다(ask). 0번은 판단과 행동을 모두 Account에게 시킵니다(tell). 잔액 규칙이 Account 안에 머무르죠.

2분 실습 — 당신 차례

아래 코드는 ask 스타일입니다. tell 스타일로 바꿔보세요: 어떤 메서드를 Order에 만들어 넣으면 될까요? 이름과 함께 저에게 보여주세요.

// 바깥(서비스)에서 주문 상태를 꺼내 판단 중
if (order.getStatus().equals("PAID") && !order.getShippedFlag()) {
  warehouse.ship(order);
  order.setStatus("SHIPPED");
}

힌트: "배송 가능한 주문인가?"라는 판단과 "배송됨으로 표시"라는 상태 변경 — 둘 다 Order가 스스로 알고 해야 할 일입니다. 목표 느낌 → if (order.isReadyToShip()) { warehouse.ship(order); order.markShipped(); }

Primary source
원출처: Robert C. Martin, Clean Code — Ch.6 "Objects and Data Structures" (객체는 행동을 노출하고 데이터를 감춘다). "Tell, Don't Ask"의 정석 해설: Martin Fowler — TellDontAsk, 냄새의 이름·처방은 Refactoring Guru — Feature Envy.
막히면 선생님(저)에게 물어보세요. "getter를 아예 쓰면 안 되나요?", "제 Order 리팩토링 봐주세요", "enum까지 꼭 써야 하나요?" — 무엇이든요. 다음 레슨은 이 흐름을 다형성으로 잇습니다: if (grade == VIP) ... else if (grade == GOLD) ... 같은 분기를 객체가 스스로 골라 행동하게 만드는 법.
