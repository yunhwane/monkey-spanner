Loop Engineering · Lesson 02

검증 신호 설계 — 믿지 말고 확인하라

Lesson 01에서 VERIFY가 루프를 에이전트로 만든다는 걸 봤습니다. 오늘은 그 VERIFY의 재료 — 검증 신호(verification signal) — 를 어떻게 고르고 만드는지 배웁니다. 끝나면 당신은 어떤 작업이든 "객관적 검증 신호"를 식별하고, 없으면 직접 만들어 루프에 붙일 수 있게 됩니다.

먼저 떠올려 보기 (1초)

Lesson 01의 장난감 루프에서 VERIFY를 담당한 함수 이름은 뭐였죠? … 그렇습니다, verify(). 오늘은 그 함수 안을 진짜로 설계합니다.

객관적 신호 vs 주관적 신호

검증 신호는 두 종류뿐입니다. 이 구분이 오늘의 전부입니다.

주관적 신호 모델의 자기보고 "다 된 것 같아요", {"done": true} → 듣기 좋지만 틀려도 그렇게 말함. 믿지 말 것. 객관적 신호 환경이 주는 사실 테스트 통과 / 컴파일 성공 / exit code 0 → 모델이 거짓말 못 함. 이걸 루프의 종료 기준으로.
오늘의 핵심 한 가지

좋은 루프는 모델의 말이 아니라 환경의 사실로 멈춘다. "끝났다"는 판단을 모델이 아니라 코드/도구가 내리게 하세요.

신호는 어디서 오나

  • 코드 작업 → 테스트 결과, 컴파일/타입체크, exit code
  • API 호출 → HTTP 상태코드, 응답 스키마 검증
  • 데이터 작업 → 행 개수, 합계 일치, assert
  • UI/브라우저 → 스크린샷, 특정 요소 존재 여부

신호가 없어 보이는 작업도 거의 항상 만들 수 있습니다: 테스트를 한 줄 쓰거나, 출력 형식을 assert 하거나, 결과를 다시 읽어 확인하는 식으로.

직접 만들어 보기 — 진짜 객관적 신호

Lesson 01 루프와 골격은 똑같지만, 이번 verify()는 모델 의견이 아니라 진짜 외부 검사(서브프로세스 exit code)를 읽습니다. 그대로 실행됩니다 — python3 verify.py.

import subprocess

def model(state):                 # 나중에 진짜 LLM으로 교체
    return state["v"] + 1         # "이번엔 이 값이 답일 것" 이라고 제안

def verify(candidate):            # ★ 객관적 신호: 환경이 판단하게 한다
    code = f"assert {candidate} % 3 == 0 and {candidate} > 0"
    r = subprocess.run(["python3", "-c", code])
    return r.returncode == 0      # exit 0 = 통과. 모델은 여기 개입 못 함.

state = {"v": 0}
for step in range(12):                       # 가드레일
    cand = model(state)
    print(f"step {step}: proposing {cand}")
    if verify(cand):                         # VERIFY = 사실 확인
        print(f"✅ {cand} 통과 (exit code 0)")
        break
    state["v"] = cand                        # 실패 → 피드백 → 다음 회차
else:
    print("⛔ 가드레일 작동")
대조: 이렇게 하면 안 된다 (주관적 신호의 함정)
def model(state):
    return {"answer": 5, "done": True}   # 모델이 스스로 끝났다고 주장

step = model(state)
if step["done"]:        # ❌ 모델 말만 믿고 종료 — 답이 틀려도 멈춘다
    print("done?")

차이가 보이나요? 위는 5가 진짜 맞는지 아무도 확인 안 합니다.

당신의 차례 (Your turn)

위 실행 코드의 검사 조건을 바꿔 직접 신호를 설계해 보세요: % 3 == 0 → % 5 == 0 으로 바꾸면 몇 회 만에, 어떤 값에서 멈출까요? 예측 → 실행 → 확인. 그리고 한 가지 더 생각해 보세요: 당신의 실제 업무 중 하나를 고른다면, 그 작업의 객관적 검증 신호는 무엇일까요? (채팅으로 알려주면 같이 루프로 만들어 봅시다.)
