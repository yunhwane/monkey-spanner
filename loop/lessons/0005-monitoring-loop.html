Loop Engineering · Lesson 05

당신의 업무를 루프로 — 로그 모니터링

드디어 장난감을 벗어납니다. 오늘은 당신이 고른 실제 업무 — 로그/모니터링 을 루프로 만듭니다. 그리고 여기엔 지금까지와 다른 종류의 루프가 등장해요. 끝나면 당신은 시계에 맞춰 도는 모니터링 루프를 설계하고, 비용을 통제할 수 있습니다.

새로운 한 가지 — 루프의 두 가지 모양
완수형 루프 (Lesson 01–04) 성공하면 STOP 예: 테스트 통과까지 고치기 모니터링 루프 (오늘) 멈추지 않고 시계로 반복 예: 1분마다 로그 점검

모니터링 루프는 "끝"이 없습니다. 종료 조건 대신 주기(interval)가 루프를 돌리고, 조건을 만족할 때만 행동합니다.

핵심 설계 비밀 — 싼 검사 먼저, 비싼 LLM은 나중

Lesson 02의 "싼 검증 먼저"가 여기서 빛납니다. 매 틱마다 Claude를 부르면 돈과 시간을 태웁니다. 대신 이렇게 합니다:

매 틱: ① 싼 검사 grep/정규식으로 ERROR 개수 센다 (LLM 0원) ② 분기 임계값 넘었나? ③ 비싼 단계 넘었을 때만 claude -p 로 조사 (이때만 과금)
오늘의 핵심 한 가지

LLM은 게이트 뒤에 둔다. 값싼 결정적 검사로 "이상한가?"를 먼저 거르고, 진짜 판단이 필요할 때만 에이전트를 깨우세요. 모니터링 루프의 비용은 거의 전부 이 설계에서 갈립니다.

직접 돌려 보기 — 이미 만들어 뒀습니다

agents/monitor.py 와 샘플 agents/app.log(ERROR 3건 포함)를 준비해 뒀어요. 한 박자 실행:

cd /Users/jeonyh/Documents/loop-engineering/agents
python3 monitor.py

ERROR가 임계값(3) 이상이라 Claude가 조사·요약까지 갑니다. 핵심 코드:

def count_errors():                  # ① 싼 검사 — LLM 없이 정규식만
    return len(re.findall(r"ERROR", open(LOG).read()))

def investigate():                   # ③ 비싼 단계 — 임계값 넘을 때만
    subprocess.run(["claude","-p", f"{LOG}의 ERROR 원인·대응 3줄 요약",
                    "--allowedTools","Read"])

def tick():                          # 루프의 '한 박자'
    n = count_errors()
    if n >= THRESHOLD:               # ② 객관적 신호로 분기
        investigate()                #    넘을 때만 과금

그럼 "반복"은 누가 시키나 — 스케줄러

tick()은 한 번만 돕니다. 반복은 바깥의 시계가 담당해요. 세 가지 방법:

  • 가장 게으른 길 — Claude Code의 /loop 스킬:
    /loop 5m python3 monitor.py  → 5분마다 자동 실행. 직접 스케줄러 안 짜도 됨.
  • OS 스케줄러 — cron: */5 * * * * cd .../agents && python3 monitor.py
  • 자체 루프 — while True: tick(); time.sleep(300) (가장 단순, 프로세스가 떠 있어야 함)

완수형 루프는 코드 안에 for/while이 있었지만, 모니터링 루프는 "반복"을 스케줄러에 위임하는 게 보통입니다.

모니터링 루프의 가드레일 (다름!)

  • 알림 폭주 방지 — 같은 에러로 매 틱 알리지 말 것. "새 에러일 때만" 또는 "마지막 알림 후 N분 경과 시만".
  • 비용 상한 — LLM 호출은 게이트 뒤에 (위에서 한 것).
  • 임계값 튜닝 — 너무 낮으면 노이즈, 너무 높으면 놓침. (Lesson 03의 가드레일 튜닝과 동일한 감각.)
