Loop Engineering · Lesson 06

알림 보내기 & 상태 기억 — 같은 에러를 두 번 알리지 마라

Lesson 05의 모니터링 루프엔 치명적 문제가 하나 있습니다. /loop나 cron으로 도는 틱은 매번 새 프로세스라 기억이 없어요. 그래서 같은 에러를 5분마다 영원히 알립니다. 오늘 끝나면 당신은 틱 사이에 상태를 기억시켜 중복 알림을 막고, 진짜 알림 채널로 내보낼 수 있습니다.

핵심 문제 — 루프는 기본적으로 기억상실증
09:00 틱 ERROR 3건 발견 → 알림 3개 "처음 보네!" 09:05 틱 (새 프로세스, 기억 없음) ERROR 3건 발견 → 또 알림 3개 "처음 보네!" ← 같은 에러를 또 09:10 틱 ... 또 ... 무한 알림 폭주 😱

해법 — 틱 사이를 잇는 durable memory

거창한 DB는 필요 없습니다. 파일 하나면 충분해요(ponytail). 본 에러의 지문(fingerprint)을 파일에 적어두고, 다음 틱엔 그걸 읽어서 새 에러만 알립니다.

오늘의 핵심 한 가지

상태 없는 틱 + 영속 파일 = 기억하는 루프. "메모리"란 거창한 게 아니라, 한 회차의 결과를 다음 회차가 읽을 수 있게 디스크에 남기는 것입니다. 이게 에이전트 메모리의 본질이에요.

지문(fingerprint) 트릭 — '같은 종류'를 같게 보기

에러 줄을 통째로 비교하면 타임스탬프 때문에 매번 "새 에러"가 됩니다. 변하는 부분(숫자·시각)을 지우고 해시하면 같은 종류의 에러는 같은 지문이 돼요.

def fingerprint(line):
    norm = re.sub(r"\d+", "#", line)      # 숫자/타임스탬프 → # 로 정규화
    return hashlib.sha1(norm.encode()).hexdigest()[:10]
# "09:03 ERROR db timeout" 과 "09:04 ERROR db timeout" → 같은 지문

직접 돌려 보기 — 두 번 실행이 포인트

agents/monitor_alert.py 를 준비해 뒀습니다. 두 번 연속 실행해 보세요:

핵심 코드:

SEEN = Path("seen.json")              # ★ durable memory

def tick():
    seen = load_seen()                # 지난 기억 읽기
    new = [e for e in errors() if fingerprint(e) not in seen]   # 새것만
    for e in new:
        notify(e.strip())             # 알림 전송
        seen.add(fingerprint(e))
    save_seen(seen)                   # 기억 저장 → 다음 틱이 읽음

정직한 한계 (ponytail:)

new를 틱 시작에 한 번에 계산하므로, 한 틱 안에서 같은 종류 에러가 여러 줄이면 그 회차엔 둘 다 알립니다(다음 틱부턴 안 함). 회차 내 중복까지 막으려면 루프 안에서 seen을 즉시 갱신·검사하세요. 지금은 "회차 간 중복 방지"가 목표라 이대로 둡니다.
